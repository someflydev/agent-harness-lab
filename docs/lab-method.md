# Lab Method

`agent-harness-lab` uses experiments, reports, and findings to improve the
harness without turning every session note into durable doctrine.

## Method

1. Start from a concrete workflow problem.
2. Run a bounded prompt, routine, script, or operator practice.
3. Record only useful facts that future sessions may need.
4. Validate the result with traceable evidence.
5. Decide whether the lesson stays local, becomes a finding, or is proposed
   for memory or doctrine promotion.
6. Avoid overfitting to one model, one provider, one tool, or one unusual run.

## Workflow Problem First

Begin with a practical question: missed closeout steps, unclear role
boundaries, noisy handoffs, weak prompt wording, validation gaps, or memory
promotion friction. A lab artifact should make a future operator or assistant
more effective.

## Bounded Trial

Use the smallest prompt, routine, or script trial that can produce evidence.
Name the setup, the variables being controlled, the validation checks, and the
stop condition. If the trial is not intentionally testing a method, treat it as
ordinary prompt execution instead of an experiment.

## Useful Facts Only

Store summaries, links, decisions, and validation evidence. Do not store raw
assistant transcripts by default. Do not preserve metadata just because it
exists. Temporary notes should remain temporary unless reviewed and promoted.

## Validation

Evidence should be traceable to prompts, files, commands, reports, commits, or
manual checks. Validation can be lightweight, but the reader should understand
why the lesson is credible and where it applies.

## Lesson Disposition

After a run or experiment, choose one disposition:

- Local: the lesson matters only to the current work.
- Report: the retrospective evidence should be preserved for review.
- Finding: the lesson describes a reviewed recurring pattern or meaningful
  workflow improvement.
- Promotion candidate: the lesson may belong in doctrine, memory, a runbook,
  or a template after explicit review.

Reports and findings support promotion decisions; they do not approve
promotion by themselves.

## Avoid Overfitting

Do not generalize from one model, one tool, one permission setting, or one
unusual repo state unless the scope says so. Prefer bounded claims and keep
provider-specific details out of doctrine unless the evidence requires them.
