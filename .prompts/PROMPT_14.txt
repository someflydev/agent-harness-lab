## Project Brief

The foundation phase should leave the repo coherent, navigable, and ready for
the implementation-hardening prompts that follow. Future automation ideas
should be documented carefully without converting the repo into a heavy runtime
prematurely.

scripts, close obvious gaps, and document a measured migration path from
human-assisted routines to helper scripts, then to richer orchestration only
after patterns prove themselves.

## Future Runtime Path Requirements

`docs/architecture/future-runtime-path.md` must describe a gradual path:

1. documented routines
2. templates and contracts
3. small helper scripts
4. structured run metadata
5. optional local indexes
6. bounded automation hooks
7. richer orchestration only after the human-assisted model is proven

Tie the path to practical signals that justify moving up the ladder.

## Traceability, Graph, And Retrieval Requirements

`docs/architecture/traceability-graph-and-semantic-retrieval.md` must explain:

- repo files and git history are source of truth
- graph-style structure may later help answer "what is connected and why?"
- semantic retrieval may later help answer "what is similar or likely relevant?"
- graph-first narrowing and vector-second enrichment may be useful later
- Neo4j and Qdrant are examples of possible future tools, not current
  requirements
- neither graph nor vector storage should become a dumping ground
- raw transcripts should not be indexed by default

Keep this as architecture guidance, not an implementation plan.
