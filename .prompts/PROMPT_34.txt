# PROMPT_34 - Assistant Driver Contracts And Capability Probes

You are implementing assistant-driver contracts for the phase-two outer loop.
Run this in a fresh coding assistant session after Prompt 33 has been
attempted.

## Project Brief

The outer loop should invoke existing assistant CLIs rather than becoming a
provider SDK. Driver contracts make that boundary inspectable: they describe
what command is expected, what capabilities are required, how to probe the
local environment safely, and which behaviors are unsupported.

The first driver layer should be conservative. It should recognize Codex,
Gemini, and Pi-style external harnesses where available, but only through local
CLI contracts and capability probes. It should not perform live model calls in
this prompt.

## Driver Contract Requirements

`docs/outer-loop/driver-contracts.md` must define a driver record with fields
such as:

- driver id
- display name
- driver kind: `subscription-cli`, `api-cli`, `external-harness`, or `manual`
- executable name
- supported invocation modes
- prompt input method: argument, stdin, prompt file, or tool-specific option
- structured output support
- final-message capture support
- sandbox or approval controls
- model selection support
- reasoning or thinking selection support
- fresh-session behavior
- resume behavior
- known limitations
- unsupported operations

The docs must distinguish between:

- a driver contract that describes a command
- a capability probe that checks local availability
- a live run that may spend quota and change files

## Registry Requirements

`registry/assistant-drivers.json` must include initial records for:

- `codex`
- `gemini`
- `pi`
- `manual`

Keep records conservative. If a capability cannot be verified from repo-local
or command-help evidence, mark it as unknown or requires verification instead
of guessing.

Do not add a `claude-subscription` external driver. If Claude is mentioned,
represent it only as a future API-backed or manual/Claude-Code-local path.

## Schema Requirements

`schemas/assistant-driver.schema.json` must validate the driver registry shape
at a lightweight level. Use simple JSON Schema constraints and avoid brittle
tool-specific validation.

## Script Requirements

Enhance `scripts/ahl.py` with driver commands such as:

```bash
python3 scripts/ahl.py driver list
python3 scripts/ahl.py driver list --json
python3 scripts/ahl.py driver check
python3 scripts/ahl.py driver probe codex --help-only --json
```

The commands should:

- read `registry/assistant-drivers.json`
- validate required driver fields
- report missing or malformed driver records
- check whether expected executables are present on `PATH` when safe
- optionally run command help probes only, not live model calls
- produce stable JSON with `ok`, `drivers`, `checks`, and `problems`

`driver probe` must default to safe inspection only. It must not send a prompt
to a model, create a session, authenticate, or write outside ordinary AHL
artifacts.

## Test Requirements

Add tests for:

- valid driver registry passes
- malformed driver registry fails with structured problems
- driver list JSON has stable fields
- unknown driver probe fails clearly
- help-only probe degrades gracefully when executable is missing

Use only standard-library tests and temporary fixtures.

## Docs Requirements

`docs/outer-loop/capability-probes.md` must explain:

- what probes check
- what probes deliberately do not check
- how to interpret missing executables
- how driver records differ from secrets or local auth state
- why live model calls are deferred to a later prompt

Update `scripts/README.md` with the new driver commands.

## Constraints

- Do not perform live model calls.
- Do not store credentials or tokens.
- Do not add provider SDKs or package dependencies.
- Do not edit local reference repos.
- Do not create a generic provider abstraction beyond local CLI driver records.
- Do not generate additional prompt files.
