# PROMPT_38 - Commit Planner And Explicit Commit Executor

You are implementing the commit-planning layer for the phase-two outer loop.
Run this in a fresh coding assistant session after Prompt 37 has been
attempted.

## Project Brief

The outer loop should be able to finish a clean prompt run by suggesting
reviewable commits with correct prefixes. It may also execute commits, but only
when explicitly authorized. This prompt adds commit planning and a cautious
commit executor that refuses to sweep unrelated work.

Commit automation is a dangerous boundary. The default must be plan-only.

## Startup Instructions

1. Inspect repo state before editing.
2. Read `AGENT.md`, `README.md`, and this prompt.
3. Read `runbooks/commit-packaging.md`, `docs/traceability.md`,

4. Run existing checks if available.
5. Check `git status --short` and preserve unrelated changes.

## Goal

Add commit planning, grouping rules, and an explicit commit executor that can
support sequential prompt batches without hiding git decisions.

## Commit Planning Requirements

`docs/outer-loop/commit-planning.md` must define grouping rules:

- default to one commit per prompt id
- allow grouping only when changes are tightly related and easy to review
- never group failed or blocked prompt work with passing prompt work
- never include unrelated pre-existing changes
- use prompt id prefixes such as `[PROMPT_38]`
- include validation evidence in commit bodies when useful
- record skipped or unavailable checks honestly
- prefer explicit file lists over broad staging

Define examples of good and bad grouping.

## Script Requirements

Enhance `scripts/ahl.py` with commands such as:

```bash
python3 scripts/ahl.py commit plan PROMPT_38 --json
python3 scripts/ahl.py commit plan --run runs/outer-loop/<run-id>/ledger.json --json
python3 scripts/ahl.py commit execute --plan runs/outer-loop/<run-id>/commit-plan.json --operator-approved
```

The planner should:

- inspect `git status --short`
- read trace/run ledger data when supplied
- identify changed files by prompt when possible
- separate untracked, modified, deleted, and staged files
- detect likely unrelated pre-existing changes when possible
- produce a commit plan artifact
- default to plan-only

The executor should:

- require an explicit approval flag such as `--operator-approved`
- stage only the specific files listed in the commit plan
- refuse `git add .` behavior
- refuse to commit if the plan references missing files
- refuse to commit if there are already staged files not in the plan
- refuse to commit if validation status is blocked or failed unless a
  `--allow-failed` style flag is explicitly supplied and documented
- use `git commit` with a generated message file, not fragile shell quoting
- record resulting commit hash in the run ledger or trace artifact when
  feasible

If executing real commits is too risky for this prompt, implement a tested
`--dry-run` executor and document the remaining work clearly. Do not pretend
real commit execution exists unless it is implemented and tested.

## Commit Message Requirements

`templates/outer-loop/commit-message.md` must support:

- subject with prompt prefix
- concise summary
- validation evidence
- changed areas
- follow-up notes or blockers

Messages should be review-oriented and avoid noisy autogenerated prose.

## Test Requirements

Add tests for:

- commit plan generation with modified and untracked files
- prompt-prefix subject generation
- executor requires explicit approval
- executor stages only listed files using a temporary git repo
- executor refuses unrelated staged files
- executor records or reports commit hash when successful
- JSON output has stable fields

Skip git-specific tests cleanly if git is unavailable.

## Constraints

- Do not push, tag, publish, or amend commits.
- Do not use `git add .` or `git add -A`.
- Do not commit unrelated user changes.
- Do not delete files automatically.
- Do not invoke assistant CLIs.
- Do not add external dependencies.
- Do not generate additional prompt files.
