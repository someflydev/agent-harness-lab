# PROMPT_02 - Doctrine, Glossary, And Anti-Patterns

You are implementing the foundational doctrine layer for `agent-harness-lab`.
Run this in a fresh coding assistant session after Prompt 01 has been attempted.

## Project Brief

This repo is a human-assisted harness lab for disciplined orchestration of
coding-agent work. It should make repeatable prompt-bounded execution,
structured handoffs, role boundaries, memory promotion, validation gates, and
operator-visible routines concrete. It should remain distinct from
`agent-context-base`, `pi-mono`, and `claw-code`, while selectively adapting
compatible ideas from each.

2. Read `AGENT.md`, `README.md`, and this prompt.
3. Read `docs/README.md`, `docs/operator-start.md`, and
   `docs/reference-influences.md` if they exist.

## Doctrine Themes To Capture

The doctrine should clearly define these core ideas:

- session boundaries are a feature, not a limitation
- the human operator is part of the harness
- prompt-bounded work is the unit of execution
- Execute -> Audit -> Preflight -> Bridge -> Reset is a core session rhythm
- completion and next-prompt readiness are distinct checks
- adjacent-prompt awareness is a disciplined middle ground between tunnel
  vision and over-planning
- repo files and git history are the source of truth
- raw agent chatter is not durable memory
- automation should follow proven human-operable routines, not precede them
- scripts should emit stable structured output when they are used for future
  automation
- graph or vector systems, if introduced later, should be derived indexes, not
  primary truth stores

## Anti-Patterns Requirements

`docs/doctrine/anti-patterns.md` must name and explain concrete failure modes,
including:

- endless session accumulation
- always-on handoff generation
- vague megaplanning
- conflating implemented with ready
- loading too many future prompts
- treating raw transcripts as durable truth
- metadata hoarding
- premature graph/vector infrastructure
- hidden autonomous behavior
- giant framework gravity
- patch-loop sprawl
- reference-repo cloning

For each anti-pattern, include the preferred corrective behavior.

## Artifact Boundary Requirements

`docs/doctrine/artifact-boundaries.md` must distinguish:

- doctrine
- runtime state
- roles
- skills
- routines
- contracts
- templates
- run manifests
- reports and findings
- examples
- temporary handoffs
- prompt-authoring artifacts
- prompt-execution artifacts

The point is to prevent the repo from becoming a memory swamp or a set of
confusing overlapping markdown buckets.

## Design Filters Requirements

`docs/doctrine/design-filters.md` should provide practical questions future
sessions can ask before adding new docs, scripts, metadata, or automation.
Include a compatibility filter for `claw-code` ideas:

- Can the useful part start as a markdown routine, checklist, template, or tiny
  script?
- Does it improve fresh-session execution or endcap quality?
- Does it work without paid API orchestration?
- Is the operator still able to inspect and override it?
- Does it preserve repo files and git history as source of truth?
