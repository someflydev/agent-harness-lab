# PROMPT_36 - Validation, Audit, And Readiness Gate Integration

You are implementing the validation gate layer for the phase-two outer loop.
Run this in a fresh coding assistant session after Prompt 35 has been
attempted.

## Project Brief

The outer runner is only useful if it can stop at the right time. After each
prompt execution, it must collect validation evidence, audit completion against
the prompt, inspect next-prompt readiness, and produce a clear status. This
prompt builds the local gate machinery that a live runner can call.

The gate cannot prove semantic correctness by itself. It should combine
deterministic checks with structured prompts/templates/skills that guide the
assistant or operator through the human judgment parts.

## Gate Model Requirements

`docs/outer-loop/gates.md` must define gate statuses:

- `pass`
- `pass-with-warnings`
- `blocked`
- `failed-validation`
- `needs-human-review`
- `driver-failed`
- `unsafe-git-state`

It must define the evidence expected after each prompt:

- prompt id
- changed files
- validation commands run
- validation outcomes
- AHL structural checks run
- completion audit status
- next-prompt readiness status
- handoff status
- commit-plan status
- stop-or-continue decision

## Script Requirements

Enhance `scripts/ahl.py` with commands such as:

```bash
python3 scripts/ahl.py outer gate PROMPT_36 --json
python3 scripts/ahl.py outer gate PROMPT_36 --plan runs/outer-loop/<run-id>/plan.json --json
```

The command should:

- inspect the current working tree through `git status --short`
- run or reference existing AHL checks when available
- check whether the prompt file exists
- check whether the immediate next prompt exists, unless this is the final
  prompt in the requested plan
- read validation commands from the plan when supplied
- record which checks were run, skipped, or unavailable
- return a structured gate report
- stop short of claiming semantic completion unless an explicit audit artifact
  exists

If running arbitrary validation commands is too risky for this prompt, implement
a conservative command allowlist or a `--record-only` mode and document the
limit clearly.

## Completion Audit Integration

`docs/outer-loop/completion-audit-integration.md` must explain:
