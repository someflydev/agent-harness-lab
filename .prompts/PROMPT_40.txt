# PROMPT_40 - Pi Adapter Experiment

You are implementing a bounded Pi adapter experiment for the phase-two outer
loop. Run this in a fresh coding assistant session after Prompt 39 has been
attempted.

## Project Brief

`pi-mono` is a richer TUI and coding-agent harness with provider/model support,
prompt templates, skills, extensions, sessions, and programmatic modes. AHL
should not clone Pi. It may, however, learn from Pi and optionally drive Pi as
an external assistant harness when the local CLI supports a safe headless or
structured mode.

This prompt should inspect the local `pi-mono` reference, document the adapter
shape, and implement only the minimum local driver support that can be tested
without real provider calls.

## Adapter Design Requirements

`docs/outer-loop/pi-adapter.md` must explain:

- which Pi modes are relevant to AHL, based on local reference docs
- how Pi differs from Codex and Gemini CLI drivers
- how Pi provider/model selection should be treated as Pi's responsibility
- what AHL should pass to Pi as prompt payload
- what AHL should expect back
- what output capture is safe
- what local capabilities must be probed before live use
- what remains unknown or unverified

Do not claim Pi support beyond what the local reference docs and implemented
probes justify.

## Comparison Requirements

`docs/outer-loop/pi-vs-ahl.md` must explain:

- Pi as an interactive/programmatic coding-agent harness
- AHL as a promptset, validation, traceability, and orchestration lab
- why AHL should avoid reimplementing Pi's TUI/provider runtime
- when using Pi as the external driver is sensible
- when direct Codex or Gemini CLI drivers are simpler

`docs/outer-loop/provider-harness-comparison.md` must compare:

- Codex subscription CLI driver
- Gemini subscription CLI driver
- Pi external harness driver
- manual driver
- future API-backed drivers

Include safety, auth, output capture, model selection, and testing differences.

## Script Requirements

Update driver logic so the `pi` driver can be:

- listed
- checked against schema
- probed safely for executable/help availability
- used in dry-run plans
- represented in live-runner plans without being invoked in tests

If the local Pi CLI shape is not reliably known, keep live invocation disabled
for `pi` and mark the driver as `manual-confirmation-required`. This is better
than guessing a command that may run incorrectly.

## Test Requirements

Add tests for:

- Pi driver registry shape
- Pi help-only probe degrades when executable is unavailable
- Pi dry-run planning works
- Pi live invocation is disabled or guarded when capability is unknown
- comparison docs are linked from outer-loop navigation

Do not invoke real Pi provider calls in tests.
